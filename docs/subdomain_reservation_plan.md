# Design & Implementation Plan: Subdomain Reservation System (v1.8.0)

This plan outlines the architecture, database schema, API logic, and step-by-step execution path to implement subdomain reservations in `lfr-tunnel`. This feature prevents naming conflicts, enforces quotas, allows random tunnel promotions, and includes email verification alerts.

---

## 1. Architectural Overview

### Reservation Status Rules
- **Regular Users:** Must reserve custom subdomains in the Portal UI before initiating a tunnel connection using that name. Tunnels connecting without a reservation will only be allowed if they request a random subdomain generated by the gateway.
- **Admin/Owner:** Can establish a tunnel using *any* subdomain without pre-registering it, provided it is not already reserved by a regular user.
- **Conflicts (Strict Block):** If a subdomain is reserved by a user, **no other user (including Admin/Owner)** can register a tunnel using that name.

### Expiration, Quotas & Quarantine
- **Lease Lifetime:** New reservations default to **7 days**. Users can request an extension through the portal. Admins/owners can approve an extension by $X$ days or make the reservation permanent.
- **Demotion:** Admin/owners can demote a permanent reservation. Demoted reservations are not deleted instantly; instead, they transition to a standard **7-day reservation**.
- **Quarantine Period:** When a subdomain reservation expires, it enters a **quarantine state** for a configurable duration (default **3 days**). During this grace period, the subdomain cannot be reserved or used by any other user, allowing the previous owner a chance to reclaim it or request an extension.
- **HTTP 410 Gone:** When an external HTTP request hits a subdomain that is currently in quarantine, the gateway will return an **HTTP 410 Gone** status code with a themed static page, indicating the service is discontinued, rather than a generic 502/404 error.
- **Quotas:** By default, users are limited to **3 reserved subdomains**. This quota is globally configurable in the YAML config and can be overridden on a per-user basis.
- **Email Notifications:** The server will trigger outbound transactional emails to inform users of reservation events (created, extended, demoted, or expired).

## 2. Database Schema Modifications

### Table: `subdomain_reservations`
Stores user-held subdomain reservations.
```sql
CREATE TABLE subdomain_reservations (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id INTEGER NOT NULL,
    subdomain TEXT NOT NULL,
    domain TEXT NOT NULL,
    expires_at DATETIME, -- NULL represents a permanent reservation
    created_at DATETIME NOT NULL,
    updated_at DATETIME NOT NULL,
    FOREIGN KEY(user_id) REFERENCES users(id) ON DELETE CASCADE
);

-- Unique index to prevent duplicate reservations across the same domain
CREATE UNIQUE INDEX idx_subdomain_domain ON subdomain_reservations(subdomain, domain);
```

### Table: `users` (New Column)
Add `max_reservations` to the `users` table to support user-specific overrides.
```sql
ALTER TABLE users ADD COLUMN max_reservations INTEGER DEFAULT NULL; -- NULL defaults to server YAML configuration
```

## 4. API Endpoints

### User Portal Endpoints
- **`GET /api/portal/reservations`**: Lists active reservations, limits, and usage stats for the authenticated user.
- **`POST /api/portal/reservations`**: Reserves a new custom subdomain for 7 days.
  - *Payload:* `{"subdomain": "myname", "domain": "lfr-demo.se"}`
  - *Validations:* Checks if user has reached reservation limits, validates subdomain format, and ensures availability.
- **`DELETE /api/portal/reservations/:id`**: Instantly releases a reservation.
- **`POST /api/portal/reservations/:id/request-extension`**: Flags a reservation as requesting an extension.
- **`POST /api/portal/reservations/promote`**: Promotes a currently active random tunnel to a permanent/7-day reservation.
  - *Payload:* `{"lease_id": "xxx"}`
  - *Validations:* Confirms the user owns the active lease, checks reservation limits, and registers the name.

### Administrative Endpoints
- **`GET /api/admin/reservations/extensions`**: Lists reservations requesting extension.
- **`POST /api/admin/reservations/:id/approve-extension`**: Updates a reservation lease.
  - *Payload:* `{"days": 30, "permanent": false}`
- **`POST /api/admin/reservations/:id/demote`**: Converts a permanent reservation back to a standard 7-day expiration.
- **`POST /api/admin/users/:id/limit`**: Overrides a user's maximum reservation quota.
  - *Payload:* `{"max_reservations": 10}`

## 8. User Interface Quota Visualization

On the new Subdomain Reservation screen:
- **Visual Progress Bar / Banner:** Displays reservation usage stats dynamically:
  - `"Reserved: N of M subdomains (M - N remaining)"`
- **Dynamic Limit Resolution:** The UI loads `max_reservations` from the user's API payload (which resolves to the user-specific override in the DB or falls back to the server YAML's `default_max_reservations`).
- **Action Triggers:** The "Reserve Subdomain" input block is dynamically disabled with a warning text if the user's quota is exhausted.

## 9. Subdomain Generator Templates

To assist users in generating subdomains, the Portal UI will offer a **"Generate Random Subdomain"** tool supporting the following selectable generator algorithms:

1. **Completely Random (Alphanumeric):**
   - *Pattern:* `[a-z0-9]{8}` (e.g., `x7a9f2bc`)
2. **What-Three-Words Style (English Word Triads):**
   - *Pattern:* `word-word-word` (e.g., `apple-banana-cherry`)
3. **Heroku/GitHub Style (Adjective-Noun-Number):**
   - *Pattern:* `adjective-noun-xxxx` (e.g., `dancing-forest-8392`, `clever-ocean-4012`)
4. **Liferay/SE-Themed Combo (Tech Adjective-Liferay Noun):**
   - *Pattern:* `tech-adjective-liferay-noun-xxx` (e.g., `hybrid-portal-918`, `headless-tomcat-554`, `cloud-extension-291`)

## 5. Logic Check-Gates (Registry & Connection Setup)

When a client makes a registration handshake (`/api/register`):
1. **Retrieve User Context:** Extract the user record from the Personal Access Token (PAT).
2. **Check Subdomain Availability & Ownership:**
   - **If Reserved by Others:** Reject with `409 Conflict`.
   - **If Reserved by Current User:** Allow connection and register active lease.
   - **If Not Reserved:**
     - **If User is Admin/Owner:** Allow connection (bypass pre-registration).
     - **If User is Regular User:**
       - **If Random Subdomain Requested:** Generate, lease, and connect.
       - **If Custom Subdomain Requested:** Reject with `403 Forbidden` ("Custom subdomains must be reserved in the portal prior to connecting").

## 6. Transactional Email Triggers

Add HTML templates and integrate mail triggers:
1. **`subdomain_reserved.html`**: Triggered when a new subdomain is successfully reserved.
2. **`extension_approved.html`**: Triggered when an admin approves an extension (displays new expiration date or permanent status).
3. **`subdomain_demoted.html`**: Sent when a permanent reservation is demoted back to a 7-day lease.
4. **`subdomain_expired.html`**: Triggered automatically by a background cleanup cron job when an un-extended reservation expires, notifying the user that the subdomain has returned to the public pool.

## 10. Security Enforcement: API Token Expiration Constraints

To enforce proper credential hygiene and encourage regular token rotation:
- **Client-Side Restriction:** In the Token Creation Modal, the `Never` option is given a specific DOM ID (`token-expiry-never`). On modal open, the script evaluates the user's role. If the user is *not* an admin or owner, the option is hidden (`display: none`), and the default dropdown value is set to `30` days.
- **Server-Side Validation:** The token generation handler (`handleCreateToken` in `pkg/server/api.go`) will enforce that `expires_in_days <= 0` (Never Expire) is strictly blocked unless the requesting user has the role `admin` or `owner`. Rejection returns a `400 Bad Request` with an explanatory JSON payload.
